# Deployment Guide

This repository contains **two websites built from one codebase**:

| Site          | Build command            | Output folder    | Goes to (subdomain)        |
| ------------- | ------------------------ | ---------------- | -------------------------- |
| ARC Lab       | `npm run build`          | `dist/`          | e.g. `arclab.itu.edu.pk`   |
| Personal site | `npm run build:personal` | `dist-personal/` | e.g. the personal subdomain |

A site flag (`VITE_SITE`) selects which one builds. The two outputs are
independent — deploying one does not affect the other.

The output is **static files** (HTML/CSS/JS only). The live server does **not**
need Node.js, npm, or any database — it just serves the files.

---

## Hosting: Hostinger (shared hosting, two subdomains)

Each site is served from the **root of its own subdomain**, e.g.
`https://arclab.itu.edu.pk/`.

Routing is handled by the included [`public/.htaccess`](public/.htaccess), which
Vite copies into the root of every build. Hostinger (Apache/LiteSpeed) reads it
automatically — **no server configuration is required.** It also handles the
single-page-app deep-link fix (so visiting `/publications` directly works
instead of returning 404).

### Recommended workflow: build locally, upload the folder

Shared hosting generally cannot run `npm`, so build on a local machine and
upload the finished folder.

**One-time, on the machine doing the build:**

1. Install [Node.js](https://nodejs.org) (LTS).
2. In the project folder: `npm install`

**Each time the site content changes:**

1. Build the site(s) you want to update:

   ```bash
   npm run build            # ARC Lab   -> dist/
   npm run build:personal   # Personal  -> dist-personal/
   ```

2. Upload the **contents** of the output folder (everything *inside* `dist/`,
   not the folder itself — including the hidden `.htaccess`) into that
   subdomain's web directory on Hostinger.

### Setting it up in Hostinger hPanel (admin)

1. **Create the subdomains.** hPanel → *Domains → Subdomains*. Create one for
   the lab and one for the personal site. Hostinger assigns each a folder, e.g.
   `public_html/arclab` (note the exact path it shows).
2. **Upload the files.** For each subdomain, put the contents of the matching
   build folder into that subdomain's directory. Three ways to do it:
   - **File Manager** (hPanel → *Files → File Manager*): open the subdomain
     folder, upload a zip of the build contents, then *Extract*. Make sure
     **hidden files are shown** so `.htaccess` is included.
   - **FTP** (an FTP client like FileZilla): drag the build contents into the
     subdomain folder. Enable "show hidden files" so `.htaccess` transfers.
   - **Git** (hPanel → *Advanced → Git*): if the admin prefers pulling from this
     repo, note that the repo holds **source**, not the built files — so this
     only helps if the build is committed or run via a build step. The
     build-locally-and-upload route above is simpler for shared hosting.
3. **Enable HTTPS.** hPanel → *Security → SSL*. Issue the free SSL certificate
   for each subdomain.

> **Important:** always upload the *contents* of `dist/` (the `index.html`,
> `assets/`, and `.htaccess`) — not the `dist` folder nested inside the web
> directory. If you see the file listing instead of the website, the files were
> placed one level too deep.

## Quick verification after deploying

- Visit the subdomain root — the site loads.
- Click around the nav (Research, Publications, etc.) — pages switch.
- **Reload the page while on a sub-page**, or open a deep link directly — it
  still loads (this confirms `.htaccess` is in place; a 404 here means the
  `.htaccess` did not get uploaded).
- After an update, a hard refresh shows the new content (old assets are cached
  by design; `index.html` is not).

## Notes

- `dist/` and `dist-personal/` are git-ignored build artifacts — they are
  regenerated by the build commands and are not stored in the repo.
- If a site is ever moved to a **sub-path** instead of a subdomain root (e.g.
  `example.com/arclab/`), the build needs Vite's `base` option set for that
  build. This is **not** the current setup; ask before changing it.
